# Digest Schema (Clean Reader — current: v2)

The schema is the real product. It makes Hermes output better in Telegram/Obsidian *and* powers the app — and because the app, the companion, and the emitter are deployed independently (and update on different clocks), the schema is a **versioned contract across drifting parts**, not just an internal format.

**Current version: `2`.** v2 is a **strictly additive** superset of v1: it adds one optional top-level field, `content` (an ordered array of typed blocks — §1.3), and changes nothing else. A v1 digest is a valid v2 digest; a v2 digest omitting `content` is indistinguishable from v1. The companion supports both (`SCHEMA_VERSIONS_SUPPORTED = (1, 2)`; `docs/architecture.md` § Companion).

Two halves:

1. **Digest contract** — what an agent POSTs to its companion.
2. **Versioning & compatibility** — how the parts stay non-breaking when they're on different versions.

---

## 1. Digest contract

The emitter kit (`docs/architecture.md` → Emitter) builds this JSON and POSTs it to the companion's ingest endpoint. The LLM cron fills the *content* (titles, prose); the **helper guarantees the envelope** — a stable unique `id`, `created_at`, `schema_version`, and required-field validation — so structure is deterministic even when the model is not.

```json
{
  "schema_version": 2,
  "id": "dgst_2026-06-29_harmony",
  "job_id": "harmony-weekly-public-digest",
  "source": "hermes-cron",
  "title": "Harmony Community Digest",
  "created_at": "2026-06-29T19:00:00-04:00",
  "urgency": "low",
  "bottom_line": "No urgent updates this week. County bylaws under review remain worth watching.",
  "content": [
    { "type": "heading", "text": "Bylaw watch" },
    { "type": "paragraph", "text": "Regional off-site levy updates remain **under review** — see [the county page](https://www.rockyview.ca/government/bylaws)." }
  ],
  "sources": [
    {
      "title": "Rocky View County Bylaws Under Review",
      "url": "https://www.rockyview.ca/government/bylaws/bylaws-under-review"
    }
  ]
}
```

A **v1** digest (still valid — the fallback for simple digests) carries `summary` / `sections` instead of `content`:

```json
{
  "schema_version": 1,
  "id": "dgst_2026-06-29_harmony",
  "...": "envelope fields as above",
  "summary": "Full prose summary of the digest...",
  "sections": [
    { "heading": "Bylaw watch", "body": "Regional off-site levy updates remain under review." }
  ]
}
```

### Field notes

- `schema_version` — integer (`1` or `2`). The companion and app use it to negotiate (see §2). Stamped by the emitter, never by the LLM.
- `id` — unique digest id (idempotency key; re-POSTing the same id updates, not duplicates). Generated by the helper, not the model, so it's stable.
- `job_id` — the source/series identifier. Used for grouping, filtering, and the per-job color stripe.
- `source` — a short string identifying the system that produced the digest (`hermes-cron`, `n8n`, `manual`). Informational; not used for routing.
- `title` — the human-readable header of the digest. Required.
- `created_at` — ISO-8601 timestamp. Used for sort order and date sectioning in the inbox.
- `urgency` — `low | normal | high | urgent`. Drives **in-app sort order** and **widget surfacing**. It is the only signal the app uses to decide whether to surface a digest more prominently.
- `bottom_line` — one-line TL;DR for the card face and the widget row. Required.
- `content[]` — **(v2, optional)** an ordered array of typed blocks the detail view renders with bespoke SwiftUI (§1.3). This is the richer body. When present, the app renders it **instead of** `summary`/`sections`.
- `summary` — **(v1 body / v2 fallback)** the main prose body. Optional; rendered in the detail view below the bottom line **when `content` is absent or has nothing renderable**.
- `sections[]` — **(v1 body / v2 fallback)** optional structured headings + body prose. Rendered in detail after the summary, same fallback condition as `summary`. Each is `{heading, body}`.
- `sources[]` — optional `{title, url}` array. Rendered as tappable links at the bottom of detail — orthogonal to the body, shown for both v1 and v2.

**Body relationship (`content` vs. `summary`/`sections`).** A digest carries **`content` OR `summary`/`sections`** (or neither). They are never rendered together: the app prefers `content` when it holds at least one renderable block (empty / whitespace-only blocks are filtered out), and falls back to `summary` + `sections` otherwise. Both bodies remain valid — `content` is for digests that want structure (metrics, callouts, ordered steps); the v1 prose body stays the right shape for a simple digest. An emitter picks one; it must not populate both expecting them to concatenate.

### 1.3 Content blocks (v2)

`content` is an ordered array of typed blocks. Each block is an object with a `"type"` discriminator; the app maps each to a bespoke SwiftUI renderer (`App/Views/ContentBlockView.swift`), styled per `docs/design-system.md`. **Six known block types:**

```json
{ "type": "paragraph", "text": "Inline **markdown** allowed here." }
{ "type": "heading",   "text": "Section heading" }
{ "type": "list", "style": "bullet", "items": ["First point", "Second point"] }
{ "type": "list", "style": "ordered", "items": ["Step one", "Step two"] }
{ "type": "callout", "variant": "warning", "title": "Heads up", "text": "Something to note." }
{ "type": "metrics", "items": [ { "label": "High", "value": "24°C" }, { "label": "Low", "value": "12°C" } ] }
{ "type": "divider" }
```

- **`paragraph`** — `{text}`. Body prose. `text` is inline markdown (see below).
- **`heading`** — `{text}`. A block-level heading within the body. Plain text.
- **`list`** — `{style?, items[]}`. `style` is `"bullet"` (default) or `"ordered"`; each item is inline markdown.
- **`callout`** — `{variant?, title?, text}`. `variant` is `"info"` (default) / `"warning"` / `"success"` / `"critical"` — **emphasis, not an action** (see below). `title` optional; `text` is inline markdown.
- **`metrics`** — `{items[]}`, each item `{label, value}` — both plain strings. Rendered as a grid of value-over-label cards.
- **`divider`** — no fields. A horizontal rule.

**Inline markdown subset.** The `text` in `paragraph` / `callout` and each `list` item may use a **restricted** inline markdown subset: `**bold**`, `*italic*`, `` `code` ``, and `[label](url)`. **No block-level markdown** — no `#` headings, `-` lists, or `>` quotes; block structure comes from the block taxonomy, not from markdown. Parsed via Foundation's `AttributedString` with `.inlineOnlyPreservingWhitespace` (`Shared/Text/CrowlyMarkdown.swift`); malformed syntax degrades to plain text rather than throwing. `heading` text and `metrics` label/value are plain strings, not markdown.

**Tolerance rules (blocks are the block-level analogue of unknown top-level fields — §2).** The "degrade-and-warn, never crash" invariant extends into the array:

- **Unknown block `type`** → preserved **verbatim** and round-trips (a v3 block survives a v2 reader, exactly as an unknown top-level field survives). The app renders the block's `text` if it has one, else skips it — never raw JSON on screen.
- **Unknown `list.style` / `callout.variant`** → falls back to the default (`bullet` / `info`).
- **Missing / wrong-typed fields degrade, never throw:** a `paragraph` missing `text` → empty → not rendered; a malformed `metrics` item (missing `label`/`value`) is dropped individually; a non-object array element or a non-array `content` value degrades to `.unknown` / `[]`. A bad block never fails the whole digest decode.

Rendering specifics (grid, callout accent bar, list markers, variant colors/symbols) live in `docs/design-system.md`.

### What's deliberately not in the schema

This is a reader. The schema describes **content shape only** — v2 blocks do not change that. There are no questions, no action items, no callbacks, no routes/intents, no `on_answer`, no `parent_question_id`, no `reply_kind`. A `callout` is **emphasis, not a button** — it has no target, no handler, no `on_tap`; it just draws the eye. If a digest wants the user to do something, it says so in prose — and the user does it in whatever tool already owns that workflow. Crowly's job ends at "the user read it."

This keeps the schema small, the companion's job small (ingest + store + serve, nothing more), and the emitter kit trivial to integrate.

## 2. Versioning & compatibility

The app, companion, and emitter ship and update independently — a user may run a months-old companion against a freshly-updated app. The contract survives this by rule, not by hope.

- **Every digest carries `schema_version`.** (v1 → v2 was the first bump: it added `content` and nothing else — the textbook additive change.)
- **Evolution is additive-only:** never remove or repurpose a field; new fields are optional with safe defaults; **unknown fields are ignored, not fatal**, on both sides. This is what keeps old-app ↔ new-companion non-breaking.
- **The companion must *preserve* unknown fields verbatim, not just tolerate them** — store the digest blob whole so a field a newer app understands survives a round-trip through an older companion. This **passthrough behavior ships in M1**, even though full version-negotiation UX (banners, "update your inbox service") is M2: a companion that strips unknowns on ingest can't be fixed later without a data migration, so the cheap insurance has to be there from the first stored digest.
- **The same preservation rule reaches *inside* `content`:** an **unknown block `type`** is the block-level analogue of an unknown top-level field — kept verbatim so a future block type authored by a newer emitter survives a round-trip through an older reader/companion (§1.3). Block evolution is additive-only too: add block types, don't repurpose the six.
- **Support window:** the app supports companion schema `N` and `N-1`. Below that, the app shows an **"update your inbox service"** banner instead of breaking.
- **Degrade-and-warn, never crash.** A field — or a block — the app doesn't recognize is hidden, not rendered as garbage.

---

## Why the contract is still the product

Three independently-deployed parts (app, companion, emitter) need a stable lingua franca, and the audience (self-hosters running their own cron) needs an emitter spec they can hit from any agent or script in any language. The schema is that lingua franca. Keeping it small, additive-only, and unknown-field-preserving is what lets Crowly be **agent-agnostic** in practice rather than just in marketing — anyone who can POST JSON can target it.
