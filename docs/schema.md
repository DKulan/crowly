# Digest + Callback Schema

The schema is the real product. It makes Hermes output better in Telegram/Obsidian *and* powers the app — and because the app, the companion, and the emitter are deployed independently (and update on different clocks), the schema is a **versioned contract across drifting parts**, not just an internal format.

Three halves:

1. **Digest contract** — what an agent POSTs to its companion.
2. **Callback contract** — how the companion records a user's answer/action, **bound to the originating job**.
3. **Versioning & compatibility** — how the parts stay non-breaking when they're on different versions.

---

## 1. Digest contract

The emitter kit (`docs/architecture.md` → Emitter) builds this JSON and POSTs it to the companion's ingest endpoint. The LLM cron fills the *content* (titles, prose, question text); the **helper guarantees the envelope** — a stable unique `id`, `created_at`, `schema_version`, and required-field validation — so structure is deterministic even when the model is not.

```json
{
  "schema_version": 1,
  "id": "dgst_2026-06-29_harmony",
  "job_id": "harmony-weekly-public-digest",
  "source": "hermes-cron",
  "title": "Harmony Community Digest",
  "created_at": "2026-06-29T19:00:00-04:00",
  "urgency": "low",
  "bottom_line": "No urgent action items. County bylaws under review remain worth watching.",
  "summary": "Full prose summary of the digest...",
  "sections": [
    { "heading": "Bylaw watch", "body": "Regional off-site levy updates remain under review." }
  ],
  "action_items": [
    {
      "id": "a1",
      "text": "Confirm Interlane EV9 drop-off window",
      "route": "task",
      "hints": { "project": "Alberta Move", "labels": ["@hermes", "@vendor"] }
    }
  ],
  "questions": [
    {
      "id": "q1",
      "text": "Should I start tracking the off-site levy bylaw as a recurring watch?",
      "reply_kind": "yes_no",
      "on_answer": {
        "yes": { "route": "task", "hints": { "text": "Set up recurring bylaw watch" } },
        "no":  { "route": "none" }
      }
    }
  ],
  "sources": [
    {
      "title": "Rocky View County Bylaws Under Review",
      "url": "https://www.rockyview.ca/government/bylaws/bylaws-under-review"
    }
  ]
}
```

### Field notes

- `schema_version` — integer. The companion and app use it to negotiate (see §3). Stamped by the emitter, never by the LLM.
- `id` — unique digest id (idempotency key; re-POSTing the same id updates, not duplicates). Generated by the helper, not the model, so it's stable.
- `job_id` — **the anchor for the whole loop.** Everything sent back references it.
- `parent_question_id` — optional. Set on a digest emitted *as the result of* a `followup` answer (the agent re-ran and produced this). It carries the `question_id` of the question the user answered, so the app can render the new digest as "in reply to your answer about X" instead of an unrelated arrival. Absent on all non-followup digests; older apps ignore it and just see a new digest — which is why it's safe to ship in v1.
- `urgency` — `low | normal | high | urgent`. Drives **in-app sort order** and **widget surfacing**. It does *not* by itself trigger a push — push is gated on open loops, not urgency (see `docs/architecture.md` → Push).
- `bottom_line` — one-line TL;DR for the card face and widget row.
- `action_items[]` / `questions[]` — each has a stable `id` so a response targets it exactly. These are what make the app a control plane, not a reader.
- `reply_kind` — `yes_no | free_text | choice`. For `choice`, include `options: ["a","b",...]`.
- `route` / `on_answer` — **intents, not tools.** See below.

### Routes are intents, not tools

A route says *what kind of thing should happen*, not *which app does it*. The companion resolves the intent against its own configured capabilities (`docs/architecture.md` → Routing).

| Intent | Meaning | Resolves to (first available wins) |
|---|---|---|
| `task` | "this is actionable" | Todoist (if configured) → companion-local markdown → stays in inbox, logged |
| `note` | "record this durably" | Obsidian-git vault (if configured) → companion-local markdown → in-inbox "saved" |
| `followup` | "hand this back to the agent" | Hermes intake (if wired) → in-inbox "needs Hermes" item |
| `none` | no side effect | — |

The schema vocabulary **never names a concrete tool** (no `todoist`, no `obsidian`). That's what lets the contract stay stable while users add or swap integrations — and what keeps the app working for someone who has Todoist but no Obsidian.

- **Action items** carry a single `route` + optional `hints` (free-form key/values the resolver may use — project, labels, due).
- **Questions** carry `on_answer`: a map from answer value → `{route, hints}`. `yes_no` keys on `"yes"`/`"no"`; `choice` keys on each option; `free_text` may carry a single default `route`. This makes a question **self-describing** — answering it triggers a routed action without the agent needing bespoke per-question code.
- Every route's **terminal fallback is "stays in the inbox, logged"** — an answer or action is never silently dropped.

> v1 deliberately has **no `create_watch` route** (an agent editing its own cron). "Yes, track this" collapses to a `task` ("Set up recurring bylaw watch") the user actions manually. This keeps the first exercised question on the simplest, safest handler and defers self-modifying infra.

## 2. Callback contract

When the user answers a question, acts on an item, or changes state, the **app POSTs to its own companion** (not to a central server). The companion executes the resolved handler locally and records the outcome. The agent is involved only for the `followup` route — and then the result returns as a *new digest*, not a chat reply.

### Question answer

```json
{
  "schema_version": 1,
  "kind": "question_answer",
  "job_id": "harmony-weekly-public-digest",
  "digest_id": "dgst_2026-06-29_harmony",
  "question_id": "q1",
  "answer": "yes",
  "note": "Track it monthly, not weekly."
}
```

### Action taken

```json
{
  "schema_version": 1,
  "kind": "action_taken",
  "job_id": "harmony-weekly-public-digest",
  "digest_id": "dgst_2026-06-29_harmony",
  "action_id": "a1",
  "overrides": { "project": "Alberta Move", "due": "2026-07-03" }
}
```

### State change (read/handled/archived)

```json
{
  "schema_version": 1,
  "kind": "state_change",
  "digest_id": "dgst_2026-06-29_harmony",
  "state": "handled"
}
```

On receiving a callback, the **companion** (not the agent) resolves the bound `on_answer`/`route` and executes it: creates the Todoist task via its own configured client, writes a durable note, flips state in its own store, or — for `followup` — posts to the user's Hermes intake. The agent never has to guess what the reply refers to, and for three of the four outcomes the agent isn't involved at all.

For `followup`, the companion returns to the app immediately (the agent runs asynchronously); when the agent finishes, it emits a **new digest** through the normal ingest path with `parent_question_id` set to the answered question's id. That closes the loop in the UI without the result ever re-entering chat.

## 3. Versioning & compatibility

The app, companion, and emitter ship and update independently — a user may run a months-old companion against a freshly-updated app. The contract survives this by rule, not by hope.

- **Every digest and callback carries `schema_version`.**
- **The companion exposes `GET /capabilities`** → `{schema_version, companion_version, supported_reply_kinds, supported_routes}`. The app calls it at pairing and on each foreground, and **renders only what this companion supports** — no dead buttons.
- **Evolution is additive-only:** never remove or repurpose a field; new fields are optional with safe defaults; **unknown fields are ignored, not fatal**, on both sides. This is what keeps old-app ↔ new-companion non-breaking.
- **The companion must *preserve* unknown fields verbatim, not just tolerate them** — store the digest blob whole so a field a newer app understands survives a round-trip through an older companion. This **passthrough behavior ships in M1**, even though full version-negotiation (`/capabilities`, banners) is M2: a companion that strips unknowns on ingest can't be fixed later without a data migration, so the cheap insurance has to be there from the first stored digest.
- **Support window:** the app supports companion schema `N` and `N-1`. Below that, the app shows an **"update your inbox service"** banner instead of breaking.
- **Degrade-and-warn, never crash.** A `reply_kind` or route the app doesn't recognize is hidden with a notice, not rendered as garbage.

---

## Why this beats chat (the core argument in one place)

In Telegram, "yes, monthly not weekly" is text the agent must map back to a question it asked hours ago, possibly interleaved with other threads. Here it arrives as `{job_id, question_id, answer}`, is resolved against the question's own `on_answer`, and is executed deterministically by the companion — unambiguous, machine-routable, and auditable. The schema *is* the differentiation.
